# Paper Reader 项目阶段总结与后续路线（交接版）

更新时间：2026-03-17

## 一、项目当前定位

这个产品的核心定位已经逐步明确：

> **从海量论文中，快速、自动化地帮用户筛出想看的论文，并按优先级排序，同时补充“是否值得精读、是否具备复现依据”的结构化信息。**

当前不是做一个“论文全文问答工具”，而是做一个：

- 主题驱动的论文推荐系统
- 对推荐结果做结构化总结
- 帮用户判断是否值得继续精读
- 帮用户提前完成“代码 / 数据 / 附录 / 实现细节”等复现证据查找
- 用推荐卡片承接，而不是让用户自己下载 PDF 后再去问 AI

---

## 二、已经达成共识的产品思路

### 1. 两层页面结构
#### 页面 1：初始推荐页 / 列表页
作用：**快速筛选**
用户打开后，应一眼知道：

- 今天推荐了哪些论文
- 哪篇优先看
- 每篇大概讲什么
- 是否相对容易上手复现

列表页不应该塞太多文字，不做全文级总结。

当前共识字段：

- 标题
- 一句话总结
- 可复现性标签
- 标签
- 排名 / 推荐优先级
- 查看详情

#### 页面 2：详情页 / 推荐卡片页
作用：**帮助用户决定是否精读**
详情页不是全文翻译，而是“稍微详细但仍然高效”的结构化推荐卡片。

共识内容：

- 推荐结论
- 研究背景与目标
- 实验方法
- 内容摘要
- 创新点
- 局限性总结
- 可复现性推荐
- 实现证据
- 复现证据信号
- 附录复现证据
- 附录证据信号
- GitHub / 数据集来源 / PDF
- 最下方单独一整行“查看原文（前往 Hugging Face Daily）”

### 2. 推荐必须有优先级
用户不希望 5 篇平铺摆出来。

共识：

- 推荐结果必须体现优先级
- 首页要能告诉用户“先看哪篇”
- 可用 Top 1 / Top 2 / Top 3 等方式呈现
- Top 1 可以有轻量高亮（如黄色五角星）
- 推荐分或推荐等级必须能支持这个排序，而不能所有都是同分

### 3. 可复现性不能靠拍脑袋
共识已经从“有无 GitHub”升级到“是否有足够复现证据”。

#### 可复现证据的来源层级
1. **直接证据**
   - GitHub
   - 代码仓库
   - 数据集来源
   - Hugging Face datasets / models
   - 项目主页
   - benchmark 页面

2. **论文内证据**
   - appendix / supplementary
   - implementation details
   - training details
   - hyperparameters
   - dataset construction
   - evaluation protocol
   - prompt template
   - 模型结构说明

3. **弱证据**
   - 使用公开 benchmark
   - 方法流程清楚
   - 实验设置相对完整
   - baseline / 指标说明充分

#### 结论
**没有 GitHub ≠ 不可复现。**
尤其是 arXiv 论文很多不会显式贴 code/data share，但会在正文或附录给出足够多的实现细节。

所以产品的价值不只是“找链接”，而是：

> **替用户完成“这篇论文到底有没有复现依据”的繁琐查找。**

## 三、当前实现过 / 讨论过的关键功能点

### 1. 数据来源与推荐流程
当前主流程是基于 Hugging Face Daily / Papers 页面抓取论文，再做主题筛选与轻分析。

目标流程：

1. 抓取 Hugging Face Daily 当天论文
2. 存入数据库
3. 用规则初筛主题候选
4. 对候选跑 `run_light_analysis`
5. 用 `passes_post_llm_topic_check` 做二次过滤
6. 保存 light analysis
7. 列表页展示推荐结果
8. 详情页展示更丰富的推荐卡片
9. 用户点击“查看原文”进入 Hugging Face Daily 原始页面
10. 用户决定是否继续精读原文 / PDF

### 2. 首页推荐不是全文问答
一直强调：
- 首页只能做“快筛”
- 详情页做“稍微详细的推荐卡片”
- 不做全文搬运，不做论文整篇翻译

### 3. 首页结果要保留返回状态
从详情页返回首页时，不能重新回到“未输入主题”的空白状态。

共识是：
- 返回首页应保留刚才的输入主题、数量、推荐列表
- 用户可以继续看其它结果，而不是重新输入

### 4. 详情页需要真正承接“当前文章”
共识：
- 点击列表页当前某篇论文，就应该至少能看到这篇文章的基础推荐信息
- 即使后端详细接口一时没有返回完整数据，也不能让用户看到空白页
- 详情页应该能先显示基础信息，再补详细分析

## 四、已经明确的优化方向

### A. 近期继续推进（优先级高）
#### 1. 详情页链路稳定
要解决的问题：

- 点击首页“查看详情”后，有时没有正确进入详情页
- 或进入详情页后，后端返回“未找到该论文的详情推荐”
- 要保证首页当前卡片和详情页之间的连接可靠
- 详情页至少要能展示当前文章的基础推荐信息
- 后端详情接口返回成功时，再补充更完整内容

#### 2. 推荐分 / 排名优先级拉开
现在存在的问题：

- 推荐分容易全是 5 分
- 导致用户会质疑为什么 Top 1 是第一
- 首页优先级缺少说服力

共识方向：

- 收紧 `calc_light_score`
- topic / novelty / reproducibility 的分桶要拉开
- recommendation（read / skim / skip）要影响分数
- “高复现”但证据不足时应扣分
- 避免所有结果都被压在最高档

#### 3. 收紧“较易快速复现”的判定标准
现在的问题：

- 太多论文被判成“较易快速复现”
- 会降低产品可信度

共识方向：

只有当下面条件至少满足两项时，才适合给 high：
- 有明确代码线索
- 有明确数据来源
- 实现细节较完整
- 训练 / 推理配置较明确

如果：
- 未检测到代码
- 未检测到数据
- 实现细节也较弱

则不应给 high，更适合给 medium。

#### 4. 复现证据模块完善
详情页应继续强化：

- GitHub
- 数据集来源
- PDF
- 实现证据
- 复现证据信号
- 附录复现证据
- 附录证据信号

目标：
- 没有就是没有
- 有就要给出真实来源
- 不能给空壳链接
- 不能敷衍推断

#### 5. PDF / 附录证据提取
这是产品亮点之一。

方向：
- 从 PDF 中找 appendix / supplementary / implementation details / training details / hyperparameters / benchmark / evaluation protocol / code release 等证据
- 把这些内容提炼成结构化字段，而不是让用户自己翻附录

---

### B. 中期优化（已经明确，但不一定立刻做）
#### 1. 详情页内容继续加厚
后续可继续优化 `light_analyzer.py` 的 prompt，使得：

- `study_goal` 更完整，不泛泛而谈
- `method_summary` 更具体，更像技术路线概述
- `content_summary` 更像“帮用户决定是否精读”的摘要
- `innovations` / `limitations` 更具体，减少空话

#### 2. 热门主题提速
当前问题：
- 输入“大语言模型”这类热门主题时，响应时间较长

已经达成的优化思路：
- 候选池控制，不必过大
- 规则先筛，再送入 LLM
- 对热门主题限制进入 LLM 的论文数量
- 不把 PDF 附录解析放到首页链路里，避免拖慢首页推荐
- 详情页再做更重的附录分析

#### 3. 更明确的推荐等级
除了数字分数，还可以有：
- 优先推荐
- 重点推荐
- 建议关注
- 可继续浏览

首页需要同时体现：
- 数字 / 分数
- 排名
- 推荐等级标签

---

### C. 后续较远优化（暂时先记住）
#### 1. RAG
已经明确是后续方向，但现在还早。

未来作用可能包括：
- 对论文内容做更细粒度检索
- 结合 PDF / 附录 / 链接做更强的问答和证据归纳
- 让详情页不只是 prompt 摘要，而是基于真实论文内容的可验证信息

#### 2. 标签树
后续希望做：
- 更细的主题标签体系
- 主题树 / 子标签树
- 让推荐更结构化，而不是只有一个用户输入词
- 也可帮助做用户偏好建模

#### 3. 用户偏好
未来方向：
- 记录用户更偏好哪些类型论文
- 更偏创新 / 更偏复现 / 更偏工程落地
- 更偏多模态 / RAG / 大语言模型 / 生物方向等

## 六、当前代码层面的重要函数与模块（必须记住）

以下是当前讨论中一直围绕的关键模块和函数。新聊天里继续调试时，应优先围绕这些函数接着看。

### 1. `main.py`
#### 关键接口 / 函数
- `run_daily_pipeline(...)`
  - 每日推荐主流程
  - 抓取论文
  - 删除当前主题旧 light analysis
  - 取候选论文
  - 规则初筛
  - 轻分析
  - 二次过滤
  - 保存结果
  - 返回首页推荐列表

- `/papers/daily/run`
  - 首页“每日推荐”对应的后端接口

- `/papers/recommendation/{paper_id}`
  - 详情页请求的后端接口
  - 当前详情页是否能成功展示，和这个接口强相关

### 2. `app/services/hf_papers_fetcher.py`
#### 关键职责
- 抓取 Hugging Face Daily / Papers 页面
- 获取 paper_url / pdf_url / arxiv_url / github_url（若能抓到）
- 尝试获取更完整的 abstract
- 后续也可能是首页提速和信息完整度的关键点

### 3. `app/services/paper_repository.py`
这是当前非常关键的总接线文件。

#### 关键函数
- `save_papers(...)`
  - 保存抓到的论文

- `get_recent_papers(...)`
  - 取候选论文池

- `save_light_analysis(...)`
  - 保存轻分析结果
  - 把完整 `analysis` 存到 `raw_json`

- `list_recommendations(...)`
  - 首页推荐列表直接来源
  - 排序、返回字段、推荐列表结构都和它强相关

- `calc_light_score(...)`
  - 当前推荐分核心
  - 后续要继续收紧和拉开差异
  - 当前“全 5 分”问题与它关系非常大

- `get_recommendation_detail(...)`
  - 详情页详细数据来源
  - 当前“详情打不开 / 查不到详情推荐”最关键函数
  - 后续新聊天里要优先继续看这里

- `_extract_real_resource_links_from_paper_page(...)`
  - 从原文页抓真实 GitHub / 数据集链接
  - 必须做到“有就是有，没有就是没有”

- `_extract_appendix_evidence_from_pdf(...)`
  - 从 PDF / 附录里提取复现证据
  - 这是当前产品亮点的重要实现点

### 4. `app/services/light_analyzer.py`
#### 关键函数 / 常量
- `LIGHT_ANALYZER_SYSTEM`
  - 当前轻分析 LLM prompt 核心
  - 后续继续优化详情质量、收紧可复现性判断、减少“全是 high”都要改这里

- `run_light_analysis(...)`
  - 首页推荐链路里的轻分析主函数

- `_normalize_result(...)`
  - 轻分析输出标准化
  - 详情页字段、证据字段是否规范也和它相关

#### 已经约定要输出 / 讨论过的重要字段
- `one_sentence_summary`
- `final_conclusion`
- `recommendation`
- `recommendation_zh`
- `topic_fit`
- `topic_fit_zh`
- `novelty_level`
- `novelty_level_zh`
- `reproducibility_level`
- `reproducibility_level_zh`
- `reproducibility_reason`
- `study_goal`
- `method_summary`
- `content_summary`
- `implementation_evidence`
- `innovations`
- `limitations`
- `evidence_signals`
- `tags`

### 5. `app/services/topic_rules.py`
#### 关键函数
- `normalize_topic_label(...)`
- `is_topic_candidate(...)`
- `passes_post_llm_topic_check(...)`

这些函数和：
- 主题是否算匹配
- 为什么结果变少
- 为什么热门主题可能又慢又严
强相关。

## 七、当前最棘手但尚未彻底解决的问题

### 1. 详情页链路仍需彻底稳定
当前已有现象：
- 点击首页“查看详情”时，体验不稳定
- 可能表现为：
  - 没有明显跳转感
  - 详情页后端返回“未找到该论文的详情推荐”
  - 缓存清掉后缺少 fallback 基础信息

### 2. 首页推荐优先级说服力不够
问题：
- 推荐分可能全是 5
- 可复现性标签可能过于宽泛
- 用户会质疑 Top 1 凭什么在第一位

### 3. 热门主题响应时间偏长
例如“大语言模型”主题：
- 候选池大
- 规则命中多
- 进 LLM 的候选多
- 导致首页等待时间偏长

## 八、下一轮新聊天建议优先解决顺序

### 第一优先级：修详情页链路
目标：
- 用户点“查看详情”后，一定能看到当前文章的基础详情
- 后端详情接口成功时，再补充完整卡片
- 不允许一片空白
- 不允许“有时行有时不行”的状态

### 第二优先级：收紧评分与可复现性判定
目标：
- 推荐分拉开层次
- 高分更有说服力
- “较易快速复现”不再泛滥
- Top 1 更可信

### 第三优先级：首页提速
目标：
- 热门主题响应时间缩短
- 但不能明显牺牲精度
- 首页尽量做轻量流程，详情页做重分析

### 第四优先级：继续增强复现证据
目标：
- GitHub / 数据集 / 附录 / 实现细节都能形成更强证据链
- 真正体现产品亮点：替用户完成复现可行性侦查

## 九、给新聊天的接续说明（可直接复制）

当前项目是一个“主题驱动的论文推荐 + 详情卡片 + 复现证据提取”系统。

我已经达成的产品共识包括：
- 首页做快速筛选
- 详情页做结构化推荐卡片，不是全文翻译
- 推荐结果必须有优先级，不允许 5 篇平铺没有主次
- 可复现性不能只看 GitHub，要看代码、数据、正文、附录、实现细节等复现证据
- 附录复现证据提取是产品亮点之一
- 后续还会做 RAG、标签树、用户偏好、本地部署给别人用等

当前最关键待解决问题：
1. 首页点“查看详情”后，详情链路不够稳定
2. 后端详情接口有时返回“未找到该论文的详情推荐”
3. 推荐分区分度不够，容易全 5 分
4. 可复现性 high 判定太宽，影响可信度
5. 热门主题响应慢，需要提速

优先继续看这些函数：
- `run_daily_pipeline`
- `run_light_analysis`
- `save_light_analysis`
- `list_recommendations`
- `get_recommendation_detail`
- `calc_light_score`
- `_extract_real_resource_links_from_paper_page`
- `_extract_appendix_evidence_from_pdf`
- `is_topic_candidate`
- `passes_post_llm_topic_check`

后续新聊天请围绕这些函数继续收口，不要重新发散产品方向。
