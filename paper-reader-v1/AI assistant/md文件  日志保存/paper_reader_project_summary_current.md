# 论文推荐系统项目阶段总结（当前对话版）

## 1. 项目目标与当前定位

本项目当前的核心目标已经明确：

- 先做一个**可用的每日论文推荐产品原型**
- 输入用户主题后，能够：
  1. 抓取当天论文
  2. 做轻分析
  3. 输出推荐卡
- 先解决**主题和论文不匹配**的问题
- 在主题匹配可靠之后，再继续做：
  - 推荐力度优化
  - 反馈机制
  - RAG
  - 标签树
  - 部署与产品化扩展

当前已经达成共识：

> 现阶段先不追求“推荐多聪明”，先保证“输入主题和输出论文必须对上”。

---

## 2. 已经打通的主链路

后端主链路已经从“抓取论文列表”进展到“能输出推荐结果”。

### 已打通接口与能力

#### 1）抓取与入库
- `GET /papers/daily/fetch`
- 能从 Hugging Face Daily Papers 抓取论文
- 能写入 SQLite

#### 2）论文列表
- `GET /papers`
- 能返回数据库中的论文列表

#### 3）轻分析
- `POST /papers/analyze_light`
- 能对论文执行轻分析
- 能把轻分析结果写入 `paper_analyses`

#### 4）推荐列表
- `GET /papers/recommendations`
- 能返回推荐结果列表
- 返回内容包含：
  - 标题
  - 一句话总结
  - 推荐语
  - recommendation / recommendation_zh
  - topic_fit / topic_fit_zh
  - novelty_level / novelty_level_zh
  - reproducibility_level / reproducibility_level_zh
  - tags
  - score

#### 5）一键产品接口
- `GET /papers/daily/run?project_topic=xxx&limit=5`
- 能自动完成：
  1. 抓取
  2. 入库
  3. 轻分析
  4. 返回推荐结果

## 3. 前端当前状态

前端已从 Swagger/手动请求，进展到最小可用产品页面。

### 已完成的前端能力

- 页面地址：`http://localhost:3000/`
- 输入框允许用户自主输入主题
- 数量输入框可控制返回篇数
- “每日推荐”按钮可触发推荐流程
- 能展示推荐卡，包括：
  - 标题
  - 一句话总结
  - 推荐语
  - 标签
  - 评分
  - 论文链接

### 已修复的前端问题

#### 1）页面自动请求问题
之前一打开页面就自动执行推荐流程，导致：
- 不点按钮也会转圈
- 后端重复分析
- 体验混乱

已达成共识：
- 页面加载时不自动请求
- 只有用户点击“每日推荐”按钮才执行

#### 2）消息提示加粗
对于以下情况，前端消息框需要突出显示：
- 今日无相关内容
- 今日仅找到几篇

已采取的处理方向：
- 当前前端消息框已支持针对“无与主题”“仅找到”文字加粗显示

## 4. 之前发现并解决/定位过的问题

### 4.1 数据库与后端结构问题

#### 已达成处理方式

- 当前数据库文件为：
  - `paper_reader.db`

### 4.2 后端接口与调用方式问题

#### 已确认
- `http://127.0.0.1:8000/...` 是 URL，不是 PowerShell 命令
- 后端接口要通过：
  - 浏览器
  - Swagger
  - curl / Invoke-RestMethod
 访问，而不是直接把 URL 当终端命令运行

---

### 4.3 轻分析与推荐排序问题

#### 已完成
- 轻分析结果已能入库
- 推荐结果已能按 `score` 返回

#### 已发现问题
- 如果完全依赖小模型判断主题相关性，容易出现：
  - 无关论文被判为相关
  - 几乎所有论文都被打高分
  - 主题筛选失真

## 5. 当前最关键问题（本轮对话核心）

当前最大问题已经明确，不是前端，也不只是模型能力，而是：

> **输入主题与输出论文匹配严重不稳定。**

### 具体表现

#### 1）同一批论文反复出现
- 输入不同主题时，返回的还是近似同一批文章

#### 2）主题不匹配
例如：
- 输入 `rag`，仍可能出现：
  - 环境类论文
  - 水质类论文
  - 纯视觉重建类论文
- 输入 `protein` 或 `antibody` 时，也可能返回明显无关内容

#### 3）大小写不一致
- `rag` 和 `RAG` 不应该返回不同结果
- 但当前出现过一个能找到、一个找不到的情况

#### 4）无结果时提示不明确
- 如果当天没有相关内容，不应显示“每日推荐流程执行完成”
- 应该明确提示：
  - 今日无与主题“xxx”严格匹配的论文
  - 或 今日仅找到 N 篇与主题“xxx”严格匹配的论文

---

## 6. 对问题的判断与共识

这一轮已经达成的最重要共识：

### 共识 1：不能一味指望提升模型性能
用户初心是：
- 尽量走小模型路线
- 降低成本
- 降低使用门槛
- 提高普及性

所以不应一开始就完全依赖更强商用模型去“暴力修正”结果。

### 共识 2：真正应该先做的是“流程协助小模型”
即：
- 通过更严格的流程设计
- 通过程序规则
- 通过硬过滤和兜底判断

让普通模型也能完成大部分任务，至少不要出现离谱返回结果。

### 共识 3：现阶段先不追求“推荐力度”，先追求“主题必须对上”
顺序必须是：

1. 先保证主题匹配可靠
2. 再优化推荐力度
3. 再做反馈、RAG、标签树

---

## 7. 已达成的技术路线（这是后续最重要的路线）

已经达成共识的正式路线如下：

### 路线总原则
不要让小模型从一堆杂乱论文里自由判断“什么相关”，而是：

> **程序先严格筛，再让小模型做精筛和解释。**

---

## 8. 新的推荐流程（已达成一致）

当前后续要落实的正确流程是：

### 第一层：主题标准化
把用户输入主题做归一化，避免：
- `rag`
- `RAG`
- ` Rag `
被当成不同主题

### 第二层：主题关键词扩展
把主题扩展成关键词集合，例如：

#### `rag`
- rag
- retrieval
- retriever
- rerank
- reranker
- vector search
- knowledge base
- document retrieval
- indexing

#### `antibody`
- antibody
- antigen
- affinity
- epitope
- antibody design
- binder
- nanobody

#### `multimodal`
- multimodal
- vision-language
- audio-visual
- cross-modal
- vision-language-action

#### `protein`
- protein
- enzyme
- folding
- structure prediction
- peptide
- amino acid

### 第三层：程序硬过滤
对于标题和摘要：
- 如果完全没有命中主题关键词
- 直接排除
- 根本不送入 LLM

### 第四层：小模型轻分析
只对已经通过硬过滤的候选论文做轻分析：
- 一句话总结
- 推荐语
- topic_fit
- recommendation
- tags

### 第五层：程序二次兜底
即使通过了轻分析，也再判断一次：
- `topic_fit == low` 直接排除
- `recommendation == skip` 直接排除
- 若 summary / tags / 标题摘要整体仍不体现主题，也排除

### 第六层：严格返回
- 不够就少返回
- 绝不为了凑数补无关论文

## 9. 当前已经开始引入/计划引入的关键机制

### 9.1 `project_topic` 绑定分析结果
已经达成共识：
- 轻分析结果必须带 `project_topic`
- 推荐查询必须按 `project_topic` 过滤
- 不能把不同主题下的推荐结果混查

### 9.2 每次运行前清掉该主题旧结果
原因：
- 以前规则不严格时，数据库里可能留下脏结果
- 如果不清理，新的严格规则也会被旧结果污染

所以新逻辑中已达成方向：
- 每次 `daily_run` 时，先清掉该主题旧的 light analyses
- 再按新规则重新生成本次结果

### 9.3 大小写统一
例如：
- `rag`
- `RAG`
- `rag `
都应归一到同一个主题标签

### 9.4 无结果/少结果明确提示
而不是统一说“执行完成”。

## 11. 各文件的职责定位（后续继续改时要遵守）

### `app/models.py`
负责数据库模型：
- `Paper`
- `PaperAnalysis`

### `app/services/light_analyzer.py`
负责轻分析：
- 调 LLM
- 规范输出
- 输出 recommendation / topic_fit / tags 等字段

### `app/services/paper_repository.py`
负责：
- 入库
- 查询
- 保存 light analysis
- 按 topic 获取 recommendations
- 删除某主题下的旧 analysis

### `app/services/topic_rules.py`
负责：
- 主题标准化
- 主题关键词扩展
- 候选论文硬过滤
- 轻分析后二次兜底校验

### `app/main.py`
负责 FastAPI 路由整合：
- daily fetch
- analyze light
- recommendations
- daily run
- pdf 深分析入口

### `app/page.tsx`
负责前端产品化页面：
- 输入主题
- 点击推荐
- 请求 daily run
- 显示推荐卡和提示信息

## 14. 目前测试中观察到的现象

### 输入 `LLM`
输出内容开始与 LLM 相关，说明：
- 主题规则开始起作用
- 但还需继续收紧

### 输入 `protein`
当前可能返回空结果，这本身是合理的  
但前端和后端提示语还需明确化

### 输入 `antibody`
当前可能仍为空，说明当天候选中未必有相关论文，或者规则已经开始阻止胡乱匹配  
这个方向是正确的

### 输入 `rag / RAG`
当前仍有明显不匹配论文混入，说明：
- 旧脏结果污染
- 主题标准化未完全落地
- RAG 规则还要更严格
